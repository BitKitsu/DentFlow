# DentFlow System Architecture

## Overview

DentFlow is a SaaS (Software as a Service) system for dental clinic management.
The architecture is based on a multi-tenant model where each clinic is a separate tenant.

## Multi-Tenant Model

Each dental clinic is a separate **tenant** in the system.
All business data is linked via `tenantId`:

```sql
-- Example: appointments are linked to a clinic
SELECT * FROM appointment WHERE tenant_id = 1;  -- DentCare Kraków
SELECT * FROM appointment WHERE tenant_id = 2;  -- SmileClinic Warszawa
```

## Authentication and Security

### JWT Token
- Token is generated by Identity Service after login
- Contains: `email`, `userId`, `tenantId`, `roles`
- Passed in header: `Authorization: Bearer <token>`
- Verified by both services

### User Roles
| Role | Description | Permissions |
|------|-------------|-------------|
| OWNER | Clinic owner | Full access: staff CRUD, clinic data, reports, schedule, all CRUD |
| DENTIST | Dentist | Appointments CRUD, patient create/edit, view rooms/catalog/schedule (read-only) |
| RECEPTIONIST | Receptionist | Appointments CRUD, patient create/edit, room create/edit, view catalog/schedule (read-only) |
| ASSISTANT | Dental assistant | Read-only: view appointments, patients, rooms, catalog, schedule |
| PATIENT | Patient | View own appointments only (`/my`) |

### Public Endpoints (no authorization)
- `/auth/register` - Registration
- `/auth/login` - Login
- `/tenants/register` - Clinic registration
- `/tenants` - Clinic list (marketplace)
- `/tenants/catalog/all` - Public service catalog
- `/tenants/staff/all` - Public staff list

## Core Service Modules

### 1. Clinic
- `Tenant` - Dental clinic
- `Location` - Branch location
- `Room` - Treatment room
- `StaffMember` - Staff (dentist, receptionist, assistant)
- `StaffWorkingHours` - Staff working hours

### 2. Patient
- `Patient` - Patient record with personal and contact data

### 3. Scheduling
- `WorkScheduleSlot` - Work schedule slot
- `Blocker` - Time blocker (vacation, inspection)

### 4. Reservation
- `Appointment` - Appointment with states: SCHEDULED -> CONFIRMED -> COMPLETED / CANCELLED / NO_SHOW
- Automatic dentist conflict detection
- In-app and email notifications after each operation

### 5. Catalog
- `ServiceCatalogItem` - Dental service (name, price, duration)

### 6. Notification
- `Notification` - In-app notification
- `EmailService` - Email sending via SendGrid

### 7. File
- `FileMetadata` - File metadata
- S3 integration (Supabase Storage)

## Data Flows

### Clinic Registration
```
1. User registers account (POST /auth/register)
2. User logs in (POST /auth/login) -> receives JWT
3. User creates clinic (POST /tenants/register)
4. Clinic is linked to user (POST /auth/tenant)
```

### Appointment Booking
```
1. Receptionist/dentist creates appointment (POST /tenants/{id}/appointments)
2. System checks for conflicts with existing dentist appointments
3. System sends confirmation email to patient
4. System creates in-app notification for dentist
5. Dentist can mark appointment as COMPLETED or CANCELLED
```

## Concurrency and Consistency

### Pessimistic Locking
Appointment conflict detection uses `SELECT FOR UPDATE` with a 5-second lock timeout:
```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@QueryHints(@QueryHint(name = "jakarta.persistence.lock.timeout", value = "5000"))
@Query("SELECT a FROM Appointment a WHERE a.tenantId = :tenantId AND a.dentistStaffId = :dentistStaffId ...")
List<Appointment> findConflictingForUpdate(...);
```

### Blocker-vs-Appointment Cross-Check
Before creating an appointment, the system checks if the time slot overlaps with any existing blocker (vacation, inspection, etc.) via `BlockerRepository.findConflicting()`.

### Database Index
`V15__add_appointment_conflict_index.sql` adds a composite index on `(tenant_id, dentist_staff_id, status, start_at, end_at)` for fast conflict lookups.

## Async Notifications

Notifications (in-app + email) are decoupled from the database transaction using Spring `ApplicationEventPublisher`:
- `AppointmentService` publishes `AppointmentCreatedEvent`, `AppointmentCancelledEvent`, `AppointmentCompletedEvent` after commit
- `AppointmentNotificationListener` handles all three events via `@Async @EventListener`
- `@EnableAsync` is configured in `CoreServiceApplication`

## Scheduled Jobs

| Job | Cron | Description |
|-----|------|-------------|
| `AppointmentNoShowJob` | `0 0 * * * *` (hourly) | Marks past SCHEDULED appointments as NO_SHOW |
| `AppointmentAutoCompleteJob` | `0 5 * * * *` (hourly at :05) | Marks past CONFIRMED appointments as COMPLETED |
